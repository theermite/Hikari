## Erreurs rencontrées → cause racine → correction

### 1. Faux « build vert » masqué par un pipe `| tail`
- **5 pourquoi** : `cargo build | tail` renvoie le code de sortie de `tail` (0), pas de
  `cargo` → un échec de compilation est passé pour un succès → détecté seulement quand
  clippy a révélé la vraie erreur `ObsString: From<&String>`.
- **Correction** : capturer le vrai code de sortie (`cmd > log 2>&1; echo EXIT=$?`), ne
  jamais lire un statut à travers un pipe.

### 2. `output.video_encoder()` inventé dans le spike
- **5 pourquoi** : API supposée de mémoire → n'existe pas sous ce nom → build `EXIT=101`.
- **Correction** : lecture du code source réel de `libobs-wrapper` → vraie API
  `get_current_video_encoder()`. Règle « vérifier à la source, jamais présumer » appliquée.

### 3. Spike B0.2 : DLL + données runtime manquantes
- **5 pourquoi** : l'exe du spike ne trouve pas `obs.dll` (pas sur le PATH) puis les
  fichiers `.effect` (data libobs absente du dossier de sortie) → le moteur panique à
  l'init.
- **Correction** : PATH vers `obs-studio/bin/64bit` au lancement + copie du `build_out`
  du spike B0.0 dans le dossier release. Runtime OBS trouvé, moteur démarré.

## Suite de session — correction méthodologique transverse

Après le dev, Jay a exprimé un **ras-le-bol transverse** (tous projets) sur la cérémonie
de process : trop de spikes/tests manuels et de **faux choix** (« je corrige ou on
s'arrête ? ») quand la suite était évidente.

**Racine** : la règle `Interpretation-Protocol` « ne pas décider qu'une action est
évidemment nécessaire » bloquait aussi la **suite évidente d'une tâche déjà demandée**,
pas seulement l'invention de travail non demandé.

**Correction (BLOQUANT)** : distinguer **portée** (le problème demandé — jamais l'élargir
sans demander) d'**exécution** (les étapes que la tâche implique déjà — corriger un bug
trouvé en la faisant, continuer une vérification). Sur l'exécution : décider et agir,
informer après.
