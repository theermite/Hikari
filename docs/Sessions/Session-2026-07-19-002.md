## Erreurs rencontrées → cause racine → correction

### 1. Test de latence B4 mesurait le mauvais chemin (finding Gate 2 Majeur)
- **5 pourquoi** : le test appelait `trigger_key()` en direct → jamais le wrapper
  `#[tauri::command]` réellement invoqué par le frontend → écart entre chemin testé et
  chemin utilisateur → un run optimise la brique telle qu'écrite, sans juger le besoin.
- **Correction** : router via le vrai wrapper (mock Tauri) **tenté puis abandonné** — bug
  Windows non résolu côté Tauri (`STATUS_ENTRYPOINT_NOT_FOUND`, issue upstream #13419).
  Repli : commentaire du test resserré sur sa portée réelle + gap documenté.

### 2. Connexion muette quand les variables d'environnement manquent (trouvé en réel par Jay)
- **5 pourquoi** : `connect_youtube` retournait tôt (variables absentes) → aucun événement
  émis → le `catch` côté interface était vide → l'écran restait sur « en attente » sans
  aucun signal → chaque étape faillible portait sa propre émission, l'une l'a oubliée.
- **Correction** : les 2 commandes passent par une **sortie d'erreur unique garantie**
  (`try_connect_*` + émission au seul point où toutes les erreurs convergent). Ferme la
  classe entière, pas un cas particulier.

### 3. Le navigateur ne s'ouvre pas au premier plan (trouvé en réel, non résolu)
- **5 pourquoi** : `explorer.exe` lance bien le navigateur (prouvé hors app) → mais la
  fenêtre ne passe pas au premier plan → Windows bloque le vol de focus par un processus
  non actif → symptôme identique Twitch + YouTube → cause racine non creusée (non bloquant).
- **Correction** : filet de sécurité — l'URL YouTube s'affiche à l'écran (comme Twitch),
  connexion possible en copiant le lien. **Cause racine reportée** (décision Jay) — noté
  dans « Bugs connus » de la note projet.

**Objectif** : rendre la connexion multi-comptes utilisable par Jay. **Processus** :
câblage → test en conditions réelles par Jay → 2 bugs remontés → correction → re-test.
**Résultat** : YouTube connecté pour de vrai, pas seulement « vert ». **Analyse** : les
deux bugs (silence sur erreur, navigateur sans focus) étaient **invisibles aux tests
automatiques** — un test headless ne voit ni un écran bloqué sur « en attente », ni une
fenêtre qui ne passe pas au premier plan. Seul l'essai réel par l'utilisateur les a fait
apparaître. **Leçon transférable** : un flux à interaction humaine (OAuth, ouverture de
fenêtre, focus) n'est prouvé que par un essai humain — « tests verts » ne couvre que la
logique, jamais le ressenti à l'écran. Prolonge la leçon B4 de la session précédente
(mesurer le trajet que l'utilisateur éprouve, pas un raccourci interne).
