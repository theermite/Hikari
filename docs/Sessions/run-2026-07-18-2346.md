## Écarts pris par rapport à la mission (avec justification)

1. **`lib.rs`/`commands.rs` touchés**, alors que le brief listait seulement
   `src/features/deck/*` et `src-tauri/src/deck_bridge.rs`. Nécessaire : Tauri
   `invoke_handler` REMPLACE (ne compose pas) — enregistrer les commandes deck sans
   fusionner l'`invoke_handler` existant aurait fait disparaître `connect_twitch` (B2b).
   Changement mécanique (visibilité + suppression d'un `register()` devenu redondant),
   zéro logique Twitch modifiée, zéro fichier `accounts/` touché.
2. **Seule la touche "automation" est câblée bout-en-bout** ; les 3 autres types de la
   maquette (bascule d'état, valeur en lecture, changement de board) ne le sont pas.
   Le PET B4 nomme "4 types de touches, palette, boards" dans son approche décidée, mais
   ces 3 autres types dépendent chacun d'une brique non construite (bascule d'état →
   scènes/audio non exposés à l'UI ; valeur en lecture → stats B-stats non fait ; board →
   UI multi-board sans contenu réel à grouper). Les fabriquer maintenant aurait produit
   soit du faux (Dignity.md : un bouton qui ne fait rien silencieusement), soit de la
   donnée statique déconnectée du moteur — contraire à Anti-Overengineering ("seulement
   ce qui est demandé ou clairement nécessaire"). Le critère d'acceptation PET B4 littéral
   ("latence < 100 ms · marche sans réseau · automation à bouton d'office sur le deck")
   est intégralement rempli et prouvé par test. Les 3 types restants sont un point
   d'extension explicite, pas un abandon : chaque brique qui les rendra réels (audio,
   scènes, B-stats) branchera son propre `DeckKey` kind sur ce même bridge.
3. **`src-tauri/crates/automation/execution.rs`** interprète les `Action` via un sink
   `deck_sink` intentionnellement no-op (documenté dans `deck_bridge.rs`) plutôt que
   d'exécuter un effet réel (changer de scène, poster au chat, ...). C'est la portée
   documentée dans `hikari-automation`'s own module doc ("Action interpretation ...
   belongs to the app that consumes this crate's Decision") — aucune brique n'a encore
   câblé ces effets réels (scènes B1a/B-shell, chat B10, audio B6). Un sink qui prétend
   agir alors qu'aucun cadran ne bouge violerait Dignity.md.
4. **`pnpm install`** a été nécessaire (worktree sans `node_modules`) avant de pouvoir
   lancer `pnpm test`/`pnpm lint` — `--frozen-lockfile`, aucun changement de version, pas
   de nouvelle dépendance externe ajoutée pour B4 (`hikari-automation` est un crate
   interne du workspace, hors périmètre ADR-010 qui gouverne les dépendances de registre).
5. **PET §7 non coché** pour B4 dans `docs/PET.md` — le brief rendait cette mise à jour
   optionnelle ; non faite ici pour rester strictement dans le scope Rust+TS demandé,
   laissée à la revue/à Jay.
